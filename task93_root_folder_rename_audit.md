# Task 93 — Root Folder Rename Audit (`readtgodot` → `mathdot`)

## Summary

The project's root folder was renamed from **`readtgodot`** to **`mathdot`**
(the repository now lives at `i:\mathdot`). This task audits every file and
folder that could be affected by that rename, fixes the references that the
rename broke, and verifies the app runs correctly afterward.

**Result:** 5 GDScript files contained hardcoded absolute paths pointing at the
old `i:/readtgodot/` location. All were updated to `i:/mathdot/`. Everything
else (installer, build scripts, Godot virtual paths, git remote) was already
folder-name-independent and needed no changes.

## What was affected

### Broken by the rename — fixed (hardcoded `i:/readtgodot/` dev fallbacks)

These are absolute "dev fallback" paths used when the app runs from the Godot
editor/binary instead of an installed copy. After the rename they pointed at a
folder that no longer exists.

| File | Line(s) | What it points at | Impact if unfixed |
|------|---------|-------------------|-------------------|
| `app/autoload/math_engine.gd` | 44 | REDUCE engine binary (`reduce.exe`) dev fallback | **Critical** — running from the editor would fail with *"Math engine binary not found"*; no CAS evaluation |
| `app/scripts/main.gd` | 91 | Screenshot capture output path | Screenshot helper writes to a dead path |
| `app/scripts/main.gd` | 219, 254, 267, 280 | Sample notebook paths (algebra/system/plotting/showcase) | Demo/keyboard shortcuts fail to open samples |
| `app/scripts/main.gd` | 377 | Task-11 screenshot output path | Screenshot helper writes to a dead path |
| `app/scripts/main.gd` | 658 | `notebooks_sample` workspace dev fallback | Auto-open of the sample workspace fails |
| `app/scripts/_libtest.gd` | 7 | Test report output | Test harness writes to a dead path |
| `app/scripts/_nbtest.gd` | 7–10 | Sample/marker/report paths | Test harness fails |
| `app/scripts/_uitest.gd` | 13–14 | Marker/report paths | Test harness fails |

All occurrences were changed `i:/readtgodot/` → `i:/mathdot/` (16 references
across the 5 files).

### NOT affected — no change needed

| Item | Why it's safe |
|------|---------------|
| `symbolic-math-workbench.iss` (installer) | Uses **relative** source paths (`#define SourceDir "app"`, `tools\reduce\...`) resolved relative to the `.iss` file location. Folder name is irrelevant. |
| `build-installer.ps1` / `build-installer.bat` | No hardcoded root-folder paths (only example `C:\path\to\ISCC.exe` placeholders). |
| `res://` paths (scenes, scripts, autoloads) | Godot virtual paths relative to the project root — independent of the OS folder name. |
| `user://` paths (font.cfg, color.cfg, packages.cfg, etc.) | Resolve to the per-user app-data folder, not the project folder. |
| Git remote | `origin → github.com/bluelight1324/symbolic-math-workbench.git`. The GitHub repo name is unrelated to the local folder name; nothing to change. |
| Installed/packaged app | Already portable — `math_engine.gd` and `main.gd` resolve paths relative to `OS.get_executable_path()` (the `exe_dir` candidate is tried **first**); the absolute path is only a secondary dev fallback. |

### Self-healing — left alone

- `app/.godot/editor/project_metadata.cfg` still contained
  `executable_path="I:/readtgodot/..."`. This file is **gitignored**
  (`app/.godot/` in `.gitignore`) and is auto-regenerated by the Godot editor,
  so it self-corrects on next open. Editing it manually is unnecessary.

## How the engine path resolution works (context for the fix)

`math_engine.gd::_engine_exe_path()` tries candidates in order:

1. `OS.get_executable_path().get_base_dir()/reduce/lib/csl/reduce.exe`
   — used by the **installed** app (engine bundled next to the binary).
2. `i:/mathdot/tools/reduce/lib/csl/reduce.exe`
   — **dev fallback** when running the raw Godot binary from `tools/godot/`.

When run from the editor, candidate 1 doesn't exist, so the app depends on
candidate 2. Because the rename invalidated candidate 2, the editor run was
silently losing its math engine until this fix.

## Note / recommendation

These absolute dev-fallback paths are inherently fragile — any future rename
or move of the project root will break them again. A more robust approach would
be to derive the dev fallback from the project root at runtime (e.g. via
`ProjectSettings.globalize_path("res://..")`) instead of hardcoding a drive
path. Out of scope for this task, but worth a future cleanup.
